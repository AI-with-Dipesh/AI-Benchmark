# v2.0.0 Known Limitations

## Overview

This document records limitations that are expected or environmental in nature. These are NOT software defects. They represent operational prerequisites or constraints that exist independent of the codebase.

## Limitations

### 1. Provider API Credentials Required

**Category**: Environmental  
**Severity**: Medium  
**Impact**: Live benchmarking and routing plan execution require valid API keys for at least one provider.

**Description**:  
The platform supports 4 providers: NVIDIA, OpenRouter, Ollama, and HuggingFace. Provider plugins are fully implemented and tested. However, without configured API keys:
- Model discovery returns empty sets
- Benchmark execution cannot proceed
- Routing plan generation reports "No eligible provider/model found"
- Live recommendations require historical benchmark data

**Mitigation**:  
Configure provider API keys in `configs/providers.yaml` or environment variables before production deployment.

**Status**: NOT A DEFECT — expected behavior without credentials.

---

### 2. Provider Quotas and Rate Limits

**Category**: Operational  
**Severity**: Low  
**Impact**: Free-tier providers enforce rate limits (HTTP 429). Production deployments should monitor quota usage.

**Description**:  
Free-tier LLM providers enforce rate limits. The platform implements:
- Retry with exponential backoff
- Rate limit detection
- Circuit breaker for failed providers
- Graceful degradation

However, sustained high-volume usage may exhaust provider quotas.

**Mitigation**:  
- Monitor provider quota usage
- Implement usage alerts
- Configure multiple fallback providers
- Use paid tiers for production workloads

**Status**: NOT A DEFECT — operational consideration.

---

### 3. Historical Data Sparsity

**Category**: Data  
**Severity**: Low  
**Impact**: Initial recommendations may be unstable with limited benchmark history.

**Description**:  
Recommendation stability improves with more historical benchmark data. With fewer than 5 runs:
- Top model may change between runs
- Confidence scores may be lower
- Trend analysis has limited data points

**Mitigation**:  
- Collect initial benchmark data before relying on recommendations
- Use fallback models when confidence is low
- Review recommendations manually during initial deployment

**Status**: NOT A DEFECT — expected behavior with sparse data.

---

### 4. Routing Execution Requires Authenticated Providers

**Category**: Environmental  
**Severity**: Medium  
**Impact**: Routing plan generation works, but execution requires live provider connections.

**Description**:  
The routing engine (`ModelSelector`) correctly generates routing plans with primary providers, fallback chains, and cost estimates. However, actual execution requires:
- Authenticated provider APIs
- Available models in provider registry
- Network connectivity to provider endpoints

**Mitigation**:  
Ensure provider credentials and network access are configured before executing benchmarks via routing.

**Status**: NOT A DEFECT — implementation complete, requires environment setup.

---

### 5. Memory Usage

**Category**: Performance  
**Severity**: Low  
**Impact**: Application uses ~245 MB memory in validation environment.

**Description**:  
The platform loads all plugins, configurations, and benchmark definitions into memory. Memory usage is stable and does not grow during operation. No memory leaks detected.

**Mitigation**:  
Monitor memory usage in production. Consider plugin lazy-loading for memory-constrained environments.

**Status**: NOT A DEFECT — acceptable for production use.
